## Project

api-obralex-py is a FastAPI REST API for **Equip Construye**, a B2B marketplace for construction materials in Lima, Peru. It serves as the backend for MCP tools consumed by equip-mcp-hub and api-maia (the agentic orchestrator).

## Commands

```bash
# Run locally
uvicorn main:app --reload --port 8000

# Format
black .

# Docker build (for Cloud Run deployment)
docker build -t api-obralex --platform linux/amd64 .
```

No test framework is configured yet.

## Architecture

```
Client (Telegram) ŌåÆ api-maia ŌåÆ equip-mcp-hub ŌåÆ api-obralex ŌåÆ Vertex AI Search + Cloud Storage
```

### Key layers

- **`src/api/`** ŌĆö FastAPI routers. Each file creates a `router` registered in `main.py` with prefix `/api/v1`.
- **`src/services/`** ŌĆö Business logic. Services are instantiated at module level in the router files (not dependency-injected).
- **`src/models/`** ŌĆö Pydantic response models.
- **`src/core/`** ŌĆö Config (env vars via `Config` class), GCP credential setup (`environment.py`), logging.

### Services

- **`VertexAISearchService`** ŌĆö Searches inventories via Google Discovery Engine. Returns `InventorySearchResult` dataclass with 45+ fields.
- **`InventorySchemaService`** ŌĆö Resolves which fields a customer must specify for a given product query. Combines Vertex AI Search (to identify category/subcategory) with a JSON schema cached from Cloud Storage (TTL-based). Resolution hierarchy: subcategory ŌåÆ category ŌåÆ default.

### Terminology

- **Inventory** = material in startup's warehouse (what we have in stock). Source of truth: MongoDB `inventories` collection, synced to BigQuery (`inventories_sanitized_prod`) and indexed in Vertex AI Search.
- **Product** = material the customer requests. Each product links to 1 inventory.
- Endpoints and code use `inventories` naming, not `products`.
- Dataset: 3,091 items, 12 categories, 49 brands. See `docs/inventories_sanitized_prod_v1.md` for full details and test queries.

### Endpoints

| Endpoint | Purpose |
|----------|---------|
| `GET /api/v1/search` | Search inventories via Vertex AI Search |
| `GET /api/v1/search/summary` | Search with AI-generated summary |
| `GET /api/v1/inventories/schema` | Get required_fields + field_options for a query |
| `POST /api/v1/schemas/reload` | Force reload schemas from GCS |
| `GET /api/v1/schemas/status` | Cache status info |

### Cloud Storage schema

The inventory schema JSON lives at `gs://{GCS_BUCKET_KNOWLEDGE}/{GCS_INVENTORY_SCHEMAS_PATH}`. It contains `subcategory_schemas` and `category_schemas` with `required_fields` and `field_options` per category. Generated by a Colab notebook analyzing the MongoDB `inventories` collection.

### Vertex AI Search setup

Both a **Datastore** (stores/indexes data from BigQuery) and an **App/Engine** (activates search, generates `servingConfig`) must exist in Vertex AI Search. The code only references the datastore ID (`VERTEX_SEARCH_DATASTORE_ID`) ŌĆö it accesses the serving config via the datastore path directly (`dataStores/{id}/servingConfigs/default_search`). The `VERTEX_SEARCH_COLLECTION` is always `default_collection`.

## Environment Variables

Defined in `src/core/config.py`. Required:

- `GOOGLE_APPLICATION_CREDENTIALS` ŌĆö path to GCP service account JSON (not needed on Cloud Run)
- `VERTEX_SEARCH_LOCATION`, `VERTEX_SEARCH_DATASTORE_ID`, `VERTEX_SEARCH_COLLECTION` ŌĆö Vertex AI Search config
- `GCS_BUCKET_KNOWLEDGE` ŌĆö Cloud Storage bucket name

Hardcoded in config: `GCP_PROJECT_ID`, `GCS_INVENTORY_SCHEMAS_PATH`, `GCS_TTL_SECONDS`.

## Deployment

Runs on **Google Cloud Run**. Cloud Run is detected via the `K_SERVICE` env var (see `src/core/environment.py`). Production uses gunicorn.
